# 定时同步多进程部署与验收清单

## 1. 模块目标

本模块用于统一管理以下同步任务的定时执行：

- 马克系统基础档案
- 马克运营账单
- 马克收款单
- 马克押金记录
- 马克预存款记录
- 金蝶基础档案

设计原则：

- 复用现有后端同步能力，不重写原有同步接口
- 定时计划使用独立管理模块维护
- 支持按计划配置目标、频率、园区范围、账簿上下文
- 同一计划下的多个同步目标以独立子进程方式执行
- 多个 FastAPI worker 场景下避免重复调度和重复执行

## 2. 运行机制

### 2.1 调度层

- 服务启动时，调度服务会先尝试获取数据库 leader 锁
- 只有获取到 leader 锁的 worker 会启动定时扫描线程
- 未获取 leader 锁的 worker 仍可响应手动“立即执行”

### 2.2 执行层

- 每条同步计划会先创建一条执行记录
- 计划中的每个目标会单独启动一个 Python 子进程执行
- 子进程内部通过统一入口复用现有同步处理函数
- 同一目标在全局范围内通过数据库 advisory lock 做并发互斥

### 2.3 并发安全

- 调度扫描使用全局 leader 锁，避免多 worker 重复扫描
- 计划领取使用数据库行锁，避免同一计划被并发领取两次
- 同步目标使用 target lock，避免同一目标在多个 worker/进程中重复执行

## 3. 部署前检查

上线前请确认以下条件：

- 数据库为 PostgreSQL
- 应用用户具备 `pg_try_advisory_lock` 和 `pg_advisory_unlock` 可用权限
- 后端运行环境可正常启动子进程
- 生产环境 Python 可执行程序与应用启动环境一致
- 现有马克、金蝶同步接口本身已可正常执行
- 管理员账号可访问定时同步管理页面

建议确认以下配置：

- FastAPI 可使用多 worker 部署
- 系统时区与业务期望一致，默认时区为 `Asia/Shanghai`
- 后端日志已接入统一采集，便于排查计划执行异常

## 4. 上线验收清单

### 4.1 基础验收

1. 打开“定时同步管理”页面，确认页面可正常加载计划列表、同步目标、项目和账簿数据。
2. 新建一条只包含 1 个目标的计划，确认可以保存成功。
3. 编辑该计划，修改执行频率和描述，确认可以更新成功。
4. 启用/停用计划，确认状态切换后 `next_run_at` 正常变化。

### 4.2 手动执行验收

1. 创建一条包含多个目标的计划。
2. 点击“立即执行”。
3. 确认生成执行记录，状态先变为 `running`，完成后进入 `success`、`partial` 或 `failed`。
4. 确认执行结果中每个目标都有独立返回结果。
5. 确认页面文案显示为“分发独立进程处理目标同步项”。

### 4.3 多进程验收

1. 使用 2 个及以上 worker 启动后端服务。
2. 配置一条短周期计划或将某计划 `next_run_at` 调整到当前时间附近。
3. 观察只有一个 worker 持续执行调度扫描。
4. 确认同一时间点同一条计划只生成 1 条执行记录。
5. 确认计划内多个目标可并发执行，且各自占用独立子进程。

### 4.4 并发互斥验收

1. 人工同时对同一计划触发两次“立即执行”。
2. 预期仅有一次领取成功，另一请求提示“Schedule is already running”。
3. 当某目标仍在运行时，另一个计划包含相同目标并发启动。
4. 预期该目标被互斥拦截，不会在全局范围内重复执行。

## 5. 异常场景验收

建议至少验证以下场景：

- 某个同步目标失败，整条计划应记录为 `partial` 或 `failed`
- 某个子进程异常退出，执行记录应保留错误信息
- 服务重启时存在 `running` 中的记录，应在恢复阶段改写为失败并释放计划状态
- 计划已到点时手动执行，完成后不应再因为同一到点窗口被自动重复补跑

## 6. 排障建议

### 6.1 页面能打开但计划不自动执行

重点检查：

- 当前 worker 是否拿到 leader 锁
- `sync_schedules.next_run_at` 是否已到期
- 计划是否启用
- 计划是否被卡在 `is_running = true`

### 6.2 手动执行成功提交，但没有结果

重点检查：

- 子进程脚本是否可正常启动
- 子进程 Python 环境是否缺少依赖
- 子进程标准输出是否返回结构化 JSON
- 原有同步函数是否在当前运行环境可直接调用

### 6.3 同一目标重复执行

重点检查：

- 数据库是否为 PostgreSQL
- advisory lock 是否可用
- 是否存在绕过调度服务、直接调用旧接口的外部脚本

## 7. 本次涉及的关键代码

- 调度服务：`backend/services/sync_schedule_service.py`
- 子进程执行入口：`backend/scripts/run_sync_schedule_target.py`
- 同步目标注册与复用入口：`backend/main.py`
- 定时同步管理页面：`frontend/src/pages/integrations/sync-schedules/index.tsx`

## 8. 建议的生产观察项

上线后建议持续观察以下内容：

- 每小时执行记录总数
- `failed` / `partial` 比例
- 单个目标平均耗时
- 是否出现长期 `running` 的计划
- 是否出现同一目标在短时间内重复启动

如果后续需要进一步增强，可以继续补：

- 执行超时控制
- 子进程最大并发数限制
- 失败重试策略
- 执行日志落表或落文件
- 按目标维度展示耗时与成功率
