# 凭证模板触发数据源扩展规范

## 1. 目标

本文档用于约束凭证模板触发条件的数据源扩展方式。

当前系统采用固定模型：

- 根触发数据源
- 普通逻辑分组 `group`
- 关联条件分组 `relation`

后续新增 OA、报表库、第三方业务库时，必须继续沿用这套模型，不允许回退到“每个普通分组都能任意切数据源”的方案。

## 2. 当前架构入口

后端核心入口：

- `backend/voucher_source_registry.py`
- `backend/main.py`

前端核心入口：

- `frontend/src/pages/vouchers/VoucherTemplates.tsx`
- `frontend/src/pages/vouchers/ConditionBuilder.tsx`
- `frontend/src/services/api.ts`
- `frontend/src/types.ts`

## 3. 核心设计原则

### 3.1 单一根触发源

每个模板只能有一个根触发数据源，例如：

- `bills`
- `receipt_bills`
- 未来的 `oa_forms`

根触发源决定：

- 模板最外层规则可直接使用哪些字段
- 哪些关联关系可被启用
- 触发预览时从哪类记录开始计算

### 3.2 普通分组不切数据源

普通分组 `group` 仅表达逻辑：

- `AND`
- `OR`

普通分组不承担数据源切换职责。

### 3.3 只有 relation 才允许跨源

如果要从根数据源跳转到别的数据源，必须使用 `relation` 节点。

`relation` 节点至少要明确：

- `resolver`
- `target_source`
- `quantifier`

其中 `resolver` 用来声明“如何从根记录找到关联记录”，系统不能靠字段名自动猜测。

## 4. 注册模型

### 4.1 模块注册

模块用于前端分组展示，例如：

- `marki`
- `oa`

定义位置：

- `backend/main.py` 中的 `MODULE_REGISTRY`

模块职责：

- 组织前端展示结构
- 标识来源归属

模块本身不参与触发执行。

### 4.2 数据源注册

定义位置：

- `backend/main.py` 中的 `SOURCE_REGISTRY`

每个数据源必须声明：

- `id`
- `module_id`
- `label`
- `source_type`
- `root_enabled`
- `field_names_builder`
- `field_options_builder`

说明：

- `root_enabled=True` 表示可作为模板根触发源
- `root_enabled=False` 表示只能作为关联目标源或占位源

示例：

- `bills`
- `receipt_bills`
- `deposit_records`
- `oa_forms`

### 4.3 关联关系注册

定义位置：

- `backend/main.py` 中的 `RELATION_REGISTRY`

每个关联关系必须声明：

- `resolver`
- `label`
- `root_source`
- `target_source`
- `loader`

`loader` 的职责是：

- 接收当前根记录
- 返回可供子条件筛选的关联记录集合

## 5. 模板 JSON 规范

模板触发条件仍然使用现有三种节点：

- `rule`
- `group`
- `relation`

示例：

```json
{
  "id": "root",
  "type": "group",
  "logic": "AND",
  "children": [
    {
      "id": "r1",
      "type": "rule",
      "field": "receipt_bills.deal_type",
      "operator": "==",
      "value": "账单实收"
    },
    {
      "id": "rel1",
      "type": "relation",
      "resolver": "receipt_to_bills",
      "target_source": "bills",
      "quantifier": "EXISTS",
      "logic": "AND",
      "children": [
        {
          "id": "r2",
          "type": "rule",
          "field": "bills.charge_item_name",
          "operator": "==",
          "value": "物业管理费"
        }
      ]
    }
  ]
}
```

## 6. 后端扩展步骤

新增一个数据源时，按以下顺序实施。

### 6.1 增加字段提供器

需要补两个函数：

- 字段名集合函数
- 字段下拉选项函数

要求：

- 不直接把前端显示文本写死在多个位置
- 字段分组逻辑要集中管理

### 6.2 注册到 SOURCE_REGISTRY

为新数据源补一条注册信息，并明确：

- 是否允许作为根触发源
- 属于哪个模块

### 6.3 补充记录加载器

如果该数据源要作为根触发源，必须补：

- 根记录加载逻辑
- 触发预览执行入口

如果该数据源暂时只是架构占位，可以先保留空字段和空加载器，但不得设置为 `root_enabled=True`。

### 6.4 增加 relation loader

如果该数据源需要关联其它来源，必须新增：

- `resolver`
- 对应的 `loader`

### 6.5 校验逻辑跟随 registry

不得新增新的硬编码白名单。

所有这些判断都必须走注册表：

- 是否支持某个 `source_type`
- 当前根源下允许哪些 `resolver`
- 子条件允许哪些字段

## 7. 前端扩展步骤

### 7.1 只消费后端元数据

前端不允许再新增写死的 relation 列表或触发源白名单。

当前应统一依赖：

- `/api/vouchers/source-modules`

该接口返回：

- `modules`
- `relations`

### 7.2 根触发源来源于 root_enabled

模板页面的“触发数据源”下拉框，只显示 `root_enabled=True` 的数据源。

### 7.3 关联条件来源于 relations

条件编辑器中“添加关联条件”的选项，只根据：

- 当前节点来源
- 后端返回的 `relations`

动态过滤生成。

## 8. 禁止事项

以下做法禁止继续引入：

- 在普通 `group` 上随意切换数据源
- 在前端硬编码新的 relation 选项
- 在后端新增新的 `if source_type in (...)` 白名单分支
- 通过字段名猜测两张表如何关联
- 让用户手工填写底层 join 条件

## 9. OA 接入约束

当前 `oa_forms` 仅为架构占位，不代表已经接入实体数据。

当前状态：

- 已有模块占位
- 已有数据源占位
- 未接入实体字段
- 未接入真实 loader
- 未接入真实 resolver
- 不允许作为根触发源

后续如果要正式接入 OA，必须至少补齐：

- `oa_forms` 字段提供器
- `oa_forms` 根记录加载器
- `oa_forms -> 目标数据源` 的关联解析器
- 对应预览和触发执行链路

## 10. 开发验收清单

新增或调整数据源后，至少验证：

1. `/api/vouchers/source-modules` 返回的 `modules` 与 `relations` 正确
2. 模板页面中根触发源下拉框展示正确
3. 关联条件下拉项只出现合法 resolver
4. 关联子条件中的字段选择已切换到目标数据源
5. 模板保存校验通过
6. 预览执行时 relation 能正确取到关联记录
7. 不影响现有 `bills / receipt_bills / deposit_records` 模板

## 11. 推荐命名规范

推荐统一命名：

- 模块：`marki`、`oa`
- 数据源：`bills`、`receipt_bills`、`deposit_records`、`oa_forms`
- 关联器：`root_to_target` 风格

例如：

- `receipt_to_bills`
- `receipt_to_deposit_collect`
- `oa_forms_to_bills`

## 12. 总结

后续所有数据源扩展都必须遵循一句话：

“根触发源负责起点，relation 负责跨源，registry 负责声明，前端只负责消费元数据。”
