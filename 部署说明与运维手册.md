# FinFlow 部署说明与运维手册

> 本文主要面向 Linux 服务器部署。  
> 如果你的环境是 Windows Server 2012，请优先查看 [部署说明_WindowsServer2012.md](/D:/FinFlow/部署说明_WindowsServer2012.md)。

本文面向 Linux 服务器部署，默认场景为：

- 前端：Vite 构建后的静态文件
- 后端：FastAPI + Uvicorn
- 反向代理：Nginx
- 进程守护：systemd
- 数据库：PostgreSQL

如果你只是本地开发，请继续使用 [start_app.bat](/D:/FinFlow/start_app.bat)。  
如果你要正式上服务器，建议按本文的生产方式部署。

## 1. 部署前确认

### 1.1 推荐环境

- 操作系统：Ubuntu 22.04 LTS 或同类 Linux 发行版
- Python：3.10 及以上
- Node.js：20 LTS
- PostgreSQL：14 及以上
- Nginx：1.18 及以上

### 1.2 本项目当前部署特点

- 后端启动时会执行 `models.Base.metadata.create_all(...)`，并自动补部分增量字段。
- 当前没有统一的 Alembic 迁移链，升级部署前建议先备份数据库。
- 前端生产环境建议固定 `VITE_API_BASE_URL=/api`，不要继续使用 `auto`。
- 后端生产环境必须关闭热重载：`APP_RELOAD=false`。

### 1.3 上线前必须处理的安全事项

- 真实 `.env` 和 `.encryption.key` 现在已经改为“不再纳入 Git 跟踪”，但它们历史上曾被跟踪过。
- 强烈建议在正式上线前轮换这些敏感信息：
  - 数据库密码
  - `MARKI_USER` / `MARKI_PASSWORD`
  - JWT `SECRET_KEY`
- 如果数据库里已经保存了加密后的外部系统凭据，服务器必须使用与当前环境一致的 `.encryption.key`，否则历史加密数据无法正常解密。

## 3. 首次部署

### 3.1 创建部署目录

```bash
sudo mkdir -p /srv/finflow
sudo chown -R $USER:$USER /srv/finflow
cd /srv/finflow
```

### 3.2 拉取代码

```bash
git clone <你的仓库地址> .
```

### 3.3 后端部署

```bash
cd /srv/finflow/backend
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

如果你未来要启用报表模块中的 SQL Server 数据源，额外补装：

```bash
pip install pyodbc
```

同时按你的系统安装 ODBC 驱动。

### 3.4 准备后端环境变量

复制示例文件：

```bash
cd /srv/finflow/backend
cp .env.example .env
```

至少填写这些项：

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `SECRET_KEY`
- `MARKI_USER`
- `MARKI_PASSWORD`
- `MARKI_SYSTEM_ID`
- `APP_HOST`
- `APP_PORT`
- `APP_RELOAD=false`
- `ALLOWED_ORIGINS`

更推荐直接配置：

```env
DATABASE_URL=postgresql+psycopg2://finflow_user:strong_password@127.0.0.1:5432/finflow
```

### 3.5 准备加密密钥

如果这是新环境且数据库里没有历史加密数据：

```bash
cd /srv/finflow/backend
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

将生成的值写入：

```bash
echo '生成出的密钥内容' > /srv/finflow/backend/.encryption.key
chmod 600 /srv/finflow/backend/.encryption.key
```

如果这是旧环境迁移：

- 不要新生成密钥
- 必须把原环境的 `backend/.encryption.key` 原样复制到服务器

### 3.6 前端部署

```bash
cd /srv/finflow/frontend
npm ci
cp .env.production.example .env.production
```

确认 `frontend/.env.production` 至少包含：

```env
VITE_API_BASE_URL=/api
```

然后构建：

```bash
npm run build
```

构建产物目录：

- [frontend/dist](/D:/FinFlow/frontend/dist)

## 5. 生产启动方式

## 5.1 systemd 启动后端

创建服务文件：

```bash
sudo nano /etc/systemd/system/finflow-backend.service
```

内容如下：

```ini
[Unit]
Description=FinFlow Backend
After=network.target postgresql.service

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/srv/finflow/backend
Environment=PYTHONUNBUFFERED=1
ExecStart=/srv/finflow/backend/.venv/bin/python -m uvicorn main:app --host 127.0.0.1 --port 8100
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable finflow-backend
sudo systemctl start finflow-backend
sudo systemctl status finflow-backend
```

说明：

- 生产环境不要用 `python main.py` 配合 `reload=True`
- 生产环境不要把后端直接暴露在公网，建议只监听 `127.0.0.1`

## 5.2 Nginx 部署前端并反代 `/api`

创建 Nginx 配置：

```bash
sudo nano /etc/nginx/sites-available/finflow
```

示例配置：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    root /srv/finflow/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8100/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

启用配置：

```bash
sudo ln -s /etc/nginx/sites-available/finflow /etc/nginx/sites-enabled/finflow
sudo nginx -t
sudo systemctl reload nginx
```

如果你使用 HTTPS，请继续接入证书，例如 `certbot`。

## 7. 数据库与迁移说明

当前项目的数据库策略是：

- 启动时自动 `create_all`
- 启动时自动补部分字段和索引
- 仓库中存在一些独立迁移脚本，但不是统一迁移框架

这意味着：

- 首次部署可由应用自动建表
- 升级部署前必须先备份数据库
- 对大版本结构调整，建议先在测试环境验证再上生产

备份示例：

```bash
pg_dump -h 127.0.0.1 -U finflow_user -d finflow > finflow_backup_$(date +%F_%H%M%S).sql
```

## 8. 常见检查项

### 8.1 后端起不来

先看服务状态：

```bash
sudo systemctl status finflow-backend
sudo journalctl -u finflow-backend -n 200 --no-pager
```

重点检查：

- `backend/.env` 是否存在
- `DATABASE_URL` 或 `DB_*` 是否正确
- `backend/.encryption.key` 是否存在
- Python 虚拟环境是否完整

### 8.2 前端打开空白页

检查：

- `frontend/dist` 是否存在
- Nginx `root` 是否指向正确目录
- `location /` 是否用了 `try_files ... /index.html`
- `VITE_API_BASE_URL` 是否为 `/api`

### 8.3 登录后接口 401/403

检查：

- `SECRET_KEY` 是否与发布前一致
- 后端是否重启成功
- 浏览器里是否残留旧 token
- 权限配置是否误收紧

### 8.4 外部系统凭据失效

重点排查：

- `.encryption.key` 是否用错
- `MARKI_USER` / `MARKI_PASSWORD` / `MARKI_SYSTEM_ID` 是否正确
- 外部服务网络是否可达

## 9. 回滚建议

### 9.1 代码回滚

```bash
cd /srv/finflow
git log --oneline -n 5
git checkout <上一个稳定版本的提交号>
```

然后重新执行：

- 后端依赖安装
- 前端构建
- 后端服务重启
- Nginx reload

### 9.2 数据回滚

如果本次升级包含结构变化或写入异常：

```bash
psql -h 127.0.0.1 -U finflow_user -d finflow < 你的备份.sql
```

## 10. 建议的上线清单

上线前建议逐项确认：

1. 代码已完成 `git status` 自检，没有误提交 `.env`、`.encryption.key`、`dist`、`__pycache__`。
2. 数据库已备份。
3. 生产 `SECRET_KEY` 已单独生成。
4. 生产 `.encryption.key` 已确认来源正确。
5. 前端已按生产配置构建。
6. 后端 `APP_RELOAD=false`。
7. Nginx 配置已通过 `nginx -t`。
8. systemd 服务能正常拉起。
9. 域名、端口、防火墙、安全组已放通。
10. 上线后完成一次登录、菜单访问、核心业务查询和外部接口连通验证。

## 11. 当前项目已知非阻断项

- 前端 `eslint` 目前仍有较多历史问题，不适合作为当前部署阻断条件。
- 后端环境尚未形成 `pytest` 发布闸门，建议后续补齐。
- 如果将来启用 SQL Server 报表数据源，需要额外安装 `pyodbc` 和系统驱动。
